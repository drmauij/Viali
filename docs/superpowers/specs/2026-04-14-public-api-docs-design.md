# Public API Docs + "API Key" Admin Tab

**Date:** 2026-04-14
**Status:** Draft — pending user review

## Goal

Split Viali's API surface into two clearly-separated UIs:

1. **In-app credentials UI** at `/admin/integrations` (tab renamed from "Leads" to **"API Key"**) — shows only hospital-specific secrets: webhook URL (with baked-in hospital ID), API key, regenerate button, enable/disable toggle, last-received telemetry. No docs.
2. **Public documentation page** at `/api` — reachable without login, designed to be shared with third-party integrators and readable by AI agents. Documents `/book` URL parameters, the leads webhook, and the conversions API. No secrets; uses `YOUR_HOSPITAL_ID` / `YOUR_API_KEY` placeholders.

A standards-compliant `/llms.txt` at the site root makes the docs auto-discoverable by AI agents.

## Non-goals

- No AI-friendly natural-language interface to `/book` — the booking-link section is a plain technical parameter reference. (An agent-friendly booking endpoint may come later as a separate project.)
- No `/llms-full.txt` (full-markdown dump). The HTML at `/api` is already scrapeable; we can add this later if agents need it.
- No multi-language docs. Public API docs stay English-only even though the app supports i18n.
- No changes to the underlying API endpoints themselves (`POST /api/webhooks/leads/:hospitalId`, `GET /api/webhooks/conversions/:hospitalId`). Only docs/UI move around.

## User-facing changes

### `/admin/integrations` — rename tab "Leads" → "API Key"

- Tab key in `validTabs` array changes to `api`. Keep `leads` as a legacy alias so existing deep links (`/admin/integrations?tab=leads`) still work.
- Tab label: **"API Key"**.
- Component `LeadWebhookCard` renamed to `ApiKeyCard` (file: `client/src/pages/admin/components/ApiKeyCard.tsx`).
- `ApiKeyCard` keeps:
  - Webhook URL (read-only input, copy button)
  - Conversions URL (read-only input, copy button)
  - API key: "Generate API Key" button → on success shows key once with show/hide + copy; subsequent loads show "Key is set (hidden for security)" + "Regenerate" button with confirm dialog
  - Enable/disable toggle with "When disabled, incoming leads will be rejected with a 403 error" helper
  - "Last lead received X ago" line when applicable
  - Status badge (Active / Disabled / Not configured)
- `ApiKeyCard` **removes**:
  - Both "Setup Instructions" and "Conversions API (read-only)" collapsibles (Meta payload, website payload, parameter tables, curl examples, response samples)
- `ApiKeyCard` **adds**:
  - A single line below the card content: `Full API documentation →` linking to `/api` (open in new tab).

### New public route `/api`

- Client-side route in `App.tsx`, registered in the "Public routes accessible regardless of auth" block so it renders for logged-out users.
- Layout: standalone docs (no Viali app sidebar / hospital switcher / top nav). A minimal top bar with the Viali logo + wordmark links back to `/`.
- Layout grid: sticky left sidebar (section nav, ~240 px) + right content pane (max-width ~800 px). Mobile: sidebar collapses into a top `<details>` nav.
- Uses Tailwind + shadcn tokens already in the app so typography and colors match. No new dependencies.

#### Content sections (in this order)

1. **Overview**
   - One-paragraph description of what the API offers.
   - Authentication model: per-hospital API keys generated by a hospital admin at `/admin/integrations` → "API Key" tab. Passed as `?key=` query param.
   - Base URL note: `https://<your-viali-host>/api/...`
   - A callout: "You'll need a hospital ID and API key. Ask your Viali admin."

2. **Booking link (`/book`)**
   - What it is: a public, tokenless (per-hospital) booking page URL.
   - URL shape: `https://<your-viali-host>/book/<HOSPITAL_BOOKING_TOKEN>`
   - Parameter tables (name · type · example · notes), grouped:
     - **Service selection**: `service`, `service_group`, `provider`
     - **Patient prefill**: `firstName`, `surname`, `email`, `phone`
     - **UTM**: `utm_source`, `utm_medium`, `utm_campaign`, `utm_term`, `utm_content`
     - **Ad click IDs**: `gclid`, `gbraid`, `wbraid`, `fbclid`, `ttclid`, `msclkid`, `igshid`, `li_fat_id`, `twclid`
     - **Misc**: `ref`, `campaign_id`, `adset_id`, `ad_id`, `promo`, `embed`
   - One full example URL with several params.

3. **Leads Webhook**
   - Endpoint: `POST /api/webhooks/leads/:hospitalId?key=YOUR_API_KEY`
   - Request headers: `Content-Type: application/json`
   - Request body fields: Required (`source`, `first_name`, `last_name`, plus one of `email`/`phone`); Meta-specific (`lead_id`, `form_id`, `operation`); Website-specific (`message`, UTMs, click IDs).
   - Response shape: `{ "status": "received", "id": "lead-uuid" }`
   - Error responses: 401 (missing/invalid key), 403 (webhook disabled), 400 (validation), 429 (rate-limited if applicable).
   - Two curl examples: website payload and Meta payload, with `YOUR_HOSPITAL_ID` / `YOUR_API_KEY` placeholders.

4. **Conversions API**
   - Endpoint: `GET /api/webhooks/conversions/:hospitalId?key=YOUR_API_KEY`
   - Query params: `platform` (`meta_forms` | `meta_ads` | `google_ads`, optional), `level` (`kept` | `surgery_planned` | `paid`, optional), `from` / `to` (ISO date, optional).
   - Conversion levels explained (same copy as current card).
   - Response: array of `{ lead_id, event_name, event_time, lead_value, currency, platform, level }`.
   - One curl example.

Every section heading has an `id` so `/api#leads-webhook` etc. works as deep links.

### New public route `/llms.txt`

- Served by the Express server as `text/plain; charset=utf-8`.
- Content (literal, committed as a static string or file):

```
# Viali API

> Viali's public API for booking links, lead ingestion, and
> ad-platform conversion reporting.

## Docs

- [API reference](/api): full human + agent docs
- [Leads webhook](/api#leads-webhook)
- [Conversions API](/api#conversions-api)
- [Booking link parameters](/api#booking-link)
```

- URLs use relative paths so the same file works across dev / staging / production hostnames.

## Technical design

### Route registration

- **Client (`client/src/App.tsx`)**: add `<Route path="/api" component={PublicApiDocs} />` inside the public-routes block (above the `!isAuthenticated` branch, next to `/book/:token`).
- **Server**: need to confirm `GET /api` (exact, no trailing segment) is not already claimed by an Express router. If it is not, no server change is needed for the HTML route — Vite's catch-all SPA handler will serve `index.html`. If it is claimed by a 404 JSON responder, we add an explicit SPA fallthrough for exact `GET /api`. To be checked during implementation.
- **Server (`/llms.txt`)**: register `app.get("/llms.txt", (req, res) => res.type("text/plain").send(LLMS_TXT))` before the SPA catch-all. Content lives in a constant or a file at `server/public/llms.txt`.

### Files

- **New**: `client/src/pages/PublicApiDocs.tsx` — the whole docs page as a single TSX file. Uses typed parameter table data (arrays of `{ name, type, example, notes }`) rendered into tables for each group.
- **Renamed**: `client/src/pages/admin/components/LeadWebhookCard.tsx` → `ApiKeyCard.tsx`. Imports updated in `Integrations.tsx`. Component body stripped of docs collapsibles, gains a "Full API documentation →" link.
- **Modified**: `client/src/pages/admin/Integrations.tsx` — rename tab key `leads` → `api`, update `validTabs` to include both (`api` preferred, `leads` legacy), update label to "API Key", update `TabsTrigger` + `TabsContent` value.
- **Modified**: `client/src/App.tsx` — add `/api` public route + lazy import of `PublicApiDocs`.
- **Modified**: `server/index.ts` (or wherever static/SPA middleware lives) — add `/llms.txt` handler.
- **New**: `server/public/llms.txt` — the literal file content above.

### Accessibility & SEO

- `<title>` on `/api`: "Viali API — documentation for third-party integrations".
- `<meta name="description">` summarizing what the API does.
- No `noindex` — we want search engines to find this.
- Each section uses `<h2>` with `id=` for anchor linking.

### i18n

None. Public docs are English-only. The page does not use the `t()` translation helper.

## Out of scope

- Signed webhooks / HMAC signatures for the leads endpoint (future hardening).
- Rate-limit documentation (none published today; add later if we enforce limits).
- OpenAPI schema / Swagger UI. The page is hand-written prose tables, not generated from a schema.
- `/llms-full.txt`. Deferred until we see agent crawlers asking for it.
- Changes to `/book/:token` behavior or parameters. Docs describe what exists today.

## Testing

- Unit / component: render `PublicApiDocs` in a Vitest test, assert each section heading and anchor id exist, assert no occurrence of an actual API key or hospital ID string (defense-in-depth against accidentally leaking a real token into docs).
- Component: render the renamed `ApiKeyCard` without any docs-collapsible content (no `Setup Instructions` text, no `Conversions API (read-only)` text).
- Server: integration test for `GET /llms.txt` → 200, `text/plain`, contains `Viali API`. Integration test for `GET /api` → 200 HTML (SPA passthrough).
- Manual: visit `/api` logged out → loads. Visit `/api#leads-webhook` → scrolls to the Leads Webhook section. Click "Full API documentation" in admin → opens `/api` in a new tab. Curl `/llms.txt` → plain text.

## Risks & mitigations

- **`/api` path conflict.** Risk: some Express middleware returns a JSON 404 for unmatched `/api/*` paths, breaking the SPA fallthrough for exact `/api`. Mitigation: during implementation, curl `GET /api` against dev server before changing anything; if it returns JSON, add an explicit SPA-passthrough route above the JSON 404 handler.
- **Deep-link breakage.** Risk: bookmarks / saved URLs to `/admin/integrations?tab=leads` break after tab rename. Mitigation: keep `leads` in `validTabs` as a silent alias that routes to the same panel.
- **Accidentally documenting secrets.** Risk: copying current `LeadWebhookCard` examples into `PublicApiDocs.tsx` could include real hospital IDs or keys if copied while a real hospital is active. Mitigation: all example URLs use the string literals `YOUR_HOSPITAL_ID` and `YOUR_API_KEY`; unit test asserts the page does not contain any UUID-shaped string.
